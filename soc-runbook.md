# SOC Analyst Runbook — SSH Brute Force & Post-Compromise Activity

**Applies to:** SSH brute force alerts, privilege escalation, backdoor account creation, suspicious downloads
**Derived from:** HackTheBox Brutus Sherlock investigation
**Analyst:** Noel Biju John | March 2026

---

## How to Use This Runbook

This runbook covers triage and response for alerts generated by the detection rules in [`detection-rules.md`](./detection-rules.md). Work alerts in sequence — each stage confirms or rules out the next. If Stage 3 (root SSH login) is confirmed, skip directly to P1 escalation.

---

## Alert 1 — SSH Brute Force: High Volume Failures from Single IP

**Rule:** SSH High Volume Failed Logins from Single IP
**Trigger:** ≥10 `Failed password` events from same `source.ip` within 60 seconds
**Severity:** Medium → escalate to High if threshold exceeds 50

### Triage Steps

1. Identify `source.ip` — external or internal?
2. Check total failure count and time window
3. Check which usernames were targeted — single account (targeted) or many accounts (wordlist scan)?
4. Check if any `Accepted password` event follows from the same IP

### Investigation Queries

**Kibana — check failure volume and targeted accounts:**
```
source.ip: "<attacker_ip>" AND event.outcome: "failure" AND process.name: "sshd"
```

**Check if success followed failures:**
```
source.ip: "<attacker_ip>" AND event.outcome: "success" AND process.name: "sshd"
```

**Splunk:**
```spl
index=linux_logs sourcetype=syslog "Failed password" src_ip="<attacker_ip>"
| stats count by src_ip, user
| sort -count
```

### Escalation Criteria
- Escalate if failure count exceeds 50 from single IP
- Escalate immediately if any `Accepted password` event found from same IP
- Escalate if root account is being targeted specifically

### Containment
- Block source IP at firewall / host-level (`iptables -A INPUT -s <ip> -j DROP`)
- Deploy Fail2Ban if not already active
- Do NOT reset credentials yet — preserve evidence first

## Alert 2 — SSH Brute Force Success: Accepted Password After Failures

**Rule:** SSH Failures Followed by Successful Login (EQL Sequence)
**Trigger:** Multiple failures → success from same `source.ip` within 5 minutes
**Severity:** High

> This alert means credentials have been compromised. Treat as **Active Incident** from this point.

### Triage Steps

1. Confirm the username that was successfully authenticated
2. Check the session timing — opened and closed within the same second? (automated tool confirming creds, not human)
3. Check for a second login from the same IP shortly after (manual interactive session)
4. Parse `wtmp` if available to confirm interactive terminal establishment

### Investigation Queries

**Kibana — find all sessions from attacker IP:**
```
source.ip: "<attacker_ip>" AND process.name: "sshd" AND event.action: "ssh_login"
```

**Splunk — session timeline:**
```spl
index=linux_logs sourcetype=syslog src_ip="<attacker_ip>" ("Accepted password" OR "session opened" OR "session closed")
| table _time, _raw
| sort _time
```

**Parse wtmp (on host):**
```bash
python3 utmp.py wtmp
# or
last -f wtmp
```

### Escalation Criteria
- Escalate immediately — root compromise is a P1
- If session opened/closed within same second: automated tool. Check for follow-up manual session
- Notify system owner and security team

### Containment
- Do NOT change root password yet — preserve evidence first
- Monitor for follow-up session from same IP
- Begin collecting auth.log and wtmp artifacts

## Alert 3 — Root Account SSH Login

**Rule:** Root Account SSH Login Detected
**Trigger:** `Accepted password for root` event from any external IP
**Severity:** High — immediate investigation required

> Root should never authenticate via SSH in a hardened environment. Any occurrence warrants investigation regardless of source IP.

### Triage Steps

1. Was this login expected? Check with system owner
2. Is the source IP internal or external?
3. Check for brute force activity preceding this login
4. Check what actions were taken during the session

### Investigation Queries

**Kibana:**
```
user.name: "root" AND event.outcome: "success" AND process.name: "sshd"
```

**Splunk — full root session activity:**
```spl
index=linux_logs sourcetype=syslog "root"
| where match(_raw, "Accepted password|session opened|session closed|useradd|usermod|sudo|COMMAND")
| table _time, _raw
| sort _time
```

### Escalation Criteria
- Escalate all root SSH logins — no exceptions
- If preceded by brute force: confirmed compromise, initiate IR

### Containment
- Disable root SSH login: `PermitRootLogin no` in `/etc/ssh/sshd_config`
- Restart SSH: `systemctl restart sshd`
- Preserve auth.log before any changes

## Alert 4 — New Local User Account Created

**Rule:** New Local User Account Created on Linux
**Trigger:** `useradd` or `adduser` executed on production server
**Severity:** Medium → High if outside change management window

### Triage Steps

1. Identify who ran `useradd` — which user, which session?
2. Was this scheduled or expected? Check change management records
3. Check if the new account was immediately added to `sudo` or `wheel` group
4. Check the timing — does this correlate with a recent SSH login from an unknown IP?

### Investigation Queries

**Kibana:**
```
process.name: ("useradd" OR "adduser") AND host.name: "<server>"
```

**Splunk:**
```spl
index=linux_logs sourcetype=syslog ("useradd" OR "new user")
| rex field=_raw "new user: name=(?P<new_user>\S+)"
| table _time, host, new_user, _raw
```

**Check if new account was added to privileged group:**
```bash
grep "usermod" auth.log | grep -i "sudo\|wheel\|admin"
```

### Escalation Criteria
- Escalate immediately if account creation was not planned
- Escalate immediately if the new account was added to `sudo`, `wheel`, or `admin` groups
- Escalate if account creation occurred within 10 minutes of an unknown SSH login

### Containment
- Lock the backdoor account: `usermod -L <username>`
- Do NOT delete yet — preserve for forensics
- Check `/etc/passwd` and `/etc/group` for all recent modifications

## Alert 5 — Account Added to Privileged Group

**Rule:** User Added to sudo / wheel / admin Group
**Trigger:** `usermod` executed with sudo/wheel/admin group argument
**Severity:** High

> Adding an account to `sudo` grants root-equivalent access. Combined with Alert 4, this confirms a backdoor with full privilege escalation.

### Triage Steps

1. Which account was modified and which group was added?
2. Who ran the `usermod` command — root? Another account?
3. Was this modification expected?
4. Is the modified account one that was just created (i.e., Alert 4 + Alert 5 together)?

### Investigation Queries

**Splunk:**
```spl
index=linux_logs sourcetype=syslog "usermod"
| rex field=_raw "add '(?P<user>\S+)' to group '(?P<group>\S+)'"
| where group IN ("sudo", "wheel", "admin", "root")
| table _time, host, user, group
```

**Verify current sudo group membership on host:**
```bash
grep sudo /etc/group
getent group sudo
```

### Escalation Criteria
- Escalate immediately — this is persistence + privilege escalation confirmed
- Combined with Alert 4: treat as confirmed backdoor installation

### Containment
- Remove account from privileged group: `gpasswd -d <username> sudo`
- Lock account: `usermod -L <username>`
- Audit all sudo activity from this account: `grep <username> /var/log/auth.log | grep sudo`

## Alert 6 — Suspicious Download via sudo curl / wget

**Rule:** sudo curl / wget to External URL
**Trigger:** `sudo /usr/bin/curl` or `sudo wget` executed with external URL
**Severity:** High → Critical if URL resolves to known malware/toolkit

### Triage Steps

1. What is the full URL? Extract from sudo COMMAND log entry
2. Submit the URL/domain to VirusTotal or URLhaus
3. Was the download successful? Check if the file exists on disk
4. Has the script been executed? Check process history (requires EDR or auditd)

### Investigation Queries

**Kibana:**
```
event.action: "sudo" AND process.args: (*curl* OR *wget*) AND process.args: (*http*)
```

**Splunk:**
```spl
index=linux_logs sourcetype=syslog "sudo" ("curl" OR "wget")
| rex field=_raw "COMMAND=(?P<cmd>.+)"
| table _time, host, user, cmd
```

**On host — check if file was downloaded and executed:**
```bash
# Check download directory
ls -la /home/<username>/
ls -la /tmp/

# Check recently executed scripts
cat /proc/<pid>/cmdline 2>/dev/null
```

### Escalation Criteria
- Escalate immediately — downloading external tools via sudo is a confirmed post-compromise action
- If URL is a known persistence toolkit (e.g., linper.sh): treat as multi-layer persistence installed

### Containment
- Block the download URL at firewall/proxy
- Delete the downloaded file if found
- Assume the script may have already executed — full system audit required
- Check for new cron jobs, SUID binaries, SSH authorized_keys modifications

## Correlation Scenario — Full SSH Brute Force to Persistence Chain

If Alerts 1 → 6 all fire from the same host within a 30-minute window:

```
Alert 1: SSH brute force from external IP               → T1110.001
Alert 2: SSH success follows failures from same IP      → T1110.001 confirmed compromise
Alert 3: Root account SSH login                         → root-level access confirmed
Alert 4: useradd executed on server                     → T1136.001 backdoor creation
Alert 5: New account added to sudo group                → T1548.003 privilege escalation
Alert 6: sudo curl to external GitHub URL               → T1105 persistence toolkit download
```

**Full chain confirmed — initiate IR immediately:**

1. Isolate the server from the network
2. Preserve auth.log, wtmp, and /var/log/syslog before any changes
3. Collect memory dump if possible
4. Identify all accounts created since the compromise time
5. Audit cron jobs, SSH authorized_keys, SUID binaries, shell profiles
6. Remove backdoor accounts, reset all credentials
7. Rebuild from clean image if full persistence scope cannot be determined

## Evidence Preservation Checklist

Before any remediation:

- [ ] Copy full `auth.log`: `cp /var/log/auth.log /evidence/auth.log.$(date +%Y%m%d)`
- [ ] Copy `wtmp`: `cp /var/log/wtmp /evidence/wtmp.$(date +%Y%m%d)`
- [ ] Parse wtmp to readable format: `python3 utmp.py wtmp > /evidence/wtmp_parsed.txt`
- [ ] Export all SIEM alert records for the incident window
- [ ] Document exact timestamps of all attacker actions
- [ ] Record all accounts created and groups modified
- [ ] Record the full sudo COMMAND entries
- [ ] Note exact timestamps of all containment actions taken

## Post-Incident Recommendations

| Priority | Recommendation |
|---|---|
| 🔴 Critical | **Disable root SSH login** — `PermitRootLogin no` in `/etc/ssh/sshd_config` |
| 🔴 Critical | **Enforce key-based SSH auth** — disable password authentication entirely |
| 🟠 High | **Deploy Fail2Ban** — auto-block IPs after ≥5 failed SSH attempts |
| 🟠 High | **Enable auditd** — captures all command executions, not just sudo |
| 🟠 High | **Alert on useradd / usermod** — new accounts on production servers should always alert |
| 🟡 Medium | **Restrict outbound curl/wget** — servers should not initiate downloads from GitHub |
| 🟡 Medium | **Monitor sudo COMMAND logs** — any curl/wget executed with sudo should alert |
| 🟡 Medium | **Regular access reviews** — audit `/etc/passwd` and `/etc/group` monthly |
